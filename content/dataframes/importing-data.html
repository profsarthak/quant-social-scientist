Importing Data: Getting It Into R

The unglamorous gate everything passes through. Reading flat files cleanly, the tibble, and why you should never trust the type-guesser.

Nothing happens until the data is in R — “if you can’t get your data into R, you can’t do data science on it!” (Wickham et al. 2023) Import is “routine and boring” 80% of the time and “weird and frustrating” the other 20%, and that 20% is where analyses silently go wrong: a column guessed as text, a decimal comma mistaken for a thousands separator. This entry is the readr toolkit for reading flat files correctly, and the one habit — pinning column types — that makes import reproducible.

Why it matters

Import is the first station of the workflow and a real technical hurdle to clear before the creative work begins. (Wickham et al. 2023) Getting it wrong corrupts everything downstream; the goal is to read the file once, correctly, reproducibly.

readr and the tibble

The readr package gives a consistent family for turning flat files into data frames: read_csv() (comma-delimited), read_csv2() (semicolon-delimited — common where the comma is the decimal mark), read_tsv() (tab-delimited). (Wickham et al. 2023) They produce a tibble — a “slightly tweaked data frame that works better in the tidyverse,” with a refined print method showing the first 10 rows and each column’s type (<dbl>, <int>, <chr>) under its name. (Wickham et al. 2023)

library(tidyverse)
students <- read_csv("students.csv")
#> Rows: 6 Columns: 5
#> ── Column specification ──────────────
#> chr (3): name, favourite.food, mealPlan
#> dbl (2): id, age          <- readr GUESSED these types

By default the first line supplies the column names; tweak with skip = n (drop leading metadata) or comment = "#" (drop comment lines). With no header, col_names = FALSE auto-names X1…Xn, or pass a character vector to set them. (Wickham et al. 2023)

WarningNever trust the type-guesser blindly

readr guesses each column’s type from the first 1000 rows. (Wickham et al. 2023) That heuristic fails when the early rows are unrepresentative — all integers at the top, doubles later; blanks before the real values. The fix the authors insist on for reproducible code: always supply col_types explicitly with helpers like col_double(), col_date(), col_factor(). When parsing does fail, problems() returns a tibble pinpointing exactly which cells and why. (Wickham et al. 2023)

read_csv("data.csv", col_types = cols(
  id   = col_integer(),
  date = col_date(format = "%Y-%m-%d"),
  rate = col_double()
))

readr vs base R’s read.csv

The readr functions are “~10× faster,” show a progress bar on big files, and avoid base R’s frustrations — they don’t silently convert strings to factors and don’t mangle column names. (Wickham et al. 2023) Prefer them.

Writing data back

write_csv() / write_tsv() save to disk — but CSV is plain text and loses type metadata, so a factor or date round-trips as a string. To preserve types exactly, write_rds() uses R’s binary format. (Wickham et al. 2023)

Many files at once

To read a folder of CSVs into one tibble, collect the paths into a vector and apply read_csv() to each with a map() (or a for loop), then bind the results — the natural job for iteration. (Wickham et al. 2023)

files <- list.files("data/", pattern = "\\.csv$", full.names = TRUE)
all   <- map(files, read_csv) |> list_rbind()

Common pitfalls

WarningWhere this goes wrong

  • Accepting guessed column types. The 1000-row heuristic misreads unrepresentative headers; pin col_types for anything you’ll rerun. (Wickham et al. 2023)

  • Wrong delimiter / decimal mark. European CSVs use ; and decimal commas — read_csv2(), not read_csv(). (Wickham et al. 2023)

  • Round-tripping types through CSV. Factors and dates degrade to text; use write_rds() to keep them. (Wickham et al. 2023)

  • Reaching for base read.csv. Slower, factor-coercing, name-munging; read_csv() avoids all three. (Wickham et al. 2023)

## readr and the tibble

The **readr** package gives a consistent family for turning flat files into data frames: **`read_csv()`**
(comma-delimited), **`read_csv2()`** (semicolon-delimited — common where the comma is the decimal mark),
**`read_tsv()`** (tab-delimited). [@wickham_r4ds] They produce a **tibble** — a *"slightly tweaked data
frame that works better in the tidyverse,"* with a refined print method showing the first 10 rows and each
column's type (`<dbl>`, `<int>`, `<chr>`) under its name. [@wickham_r4ds]

```r
library(tidyverse)
students <- read_csv("students.csv")
#> Rows: 6 Columns: 5
#> ── Column specification ──────────────
#> chr (3): name, favourite.food, mealPlan
#> dbl (2): id, age          <- readr GUESSED these types
```

By default the **first line supplies the column names**; tweak with `skip = n` (drop leading metadata) or
`comment = "#"` (drop comment lines). With no header, `col_names = FALSE` auto-names `X1…Xn`, or pass a
character vector to set them. [@wickham_r4ds]

::: {.callout-warning title="Never trust the type-guesser blindly"}
readr **guesses** each column's type from the **first 1000 rows.** [@wickham_r4ds] That heuristic *fails*
when the early rows are unrepresentative — all integers at the top, doubles later; blanks before the real
values. The fix the authors insist on for reproducible code: **always supply `col_types`** explicitly with
helpers like `col_double()`, `col_date()`, `col_factor()`. When parsing does fail, **`problems()`** returns a
tibble pinpointing exactly which cells and why. [@wickham_r4ds]
:::

```r
read_csv("data.csv", col_types = cols(
  id   = col_integer(),
  date = col_date(format = "%Y-%m-%d"),
  rate = col_double()

```
