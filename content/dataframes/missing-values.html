Missing Values

The most consequential cell in any dataset is often the empty one. What NA means, why it’s contagious, and the difference between an absence that’s flagged and one that’s simply not there.

Real data has holes. How you handle them quietly decides your results — drop the wrong rows and you bias the sample; ignore an NA and a whole summary collapses to NA. R is deliberately strict here: a missing value is unknown, and the unknown is contagious. This entry covers what NA means, the crucial split between explicit and implicit missingness, and the tools to handle each.

Why it matters

Missingness is not a formatting nuisance; it is a substantive feature of the data with its own statistics. Where the holes are, and why, can bias every estimate downstream — the same selection logic that breaks censored survival data or an experiment with attrition. R forces you to confront it rather than letting it pass silently, which is a feature, not an annoyance. (Wickham et al. 2023)

NA is contagious

In R, NA represents an unknown value. (Wickham et al. 2023) And because it’s unknown, almost any operation touching it is also unknown — “NA is contagious.” (Wickham et al. 2023)

NA > 5      # NA  — is an unknown number bigger than 5? unknown.
10 == NA    # NA  — does 10 equal an unknown? unknown.
NA == NA    # NA  — are two unknowns equal? STILL unknown (the confusing one)

That last line is the trap: NA == NA is NA, not TRUE. So you can never test for missingness with == NA; you must use is.na(). (Wickham et al. 2023)

Explicit vs implicit missingness

A subtle and important distinction: (Wickham et al. 2023)

  • An explicit missing value is “the presence of an absence” — a cell flagged NA. You can see it.
  • An implicit missing value is “the absence of a presence” — a row that simply isn’t in the table. A respondent-quarter with no record; a product-month with no sales row. You can’t see it, and it’s the more dangerous kind because a naive count treats “no row” as if the case didn’t exist.

Much of careful data work is making implicit missings explicit so you can decide what to do with them.

Tools

Summaries propagate NA by default — mean(), sum(), median() return NA if any input is missing, which is R insisting you make a choice. Pass na.rm = TRUE to drop missings before computing. (Wickham et al. 2023) For wrangling:

Tool What it does
is.na(x) the only correct test for missingness (logical vector)
coalesce(x, y, …) take the first non-missing value across several vectors — fill from a backup
tidyr::complete() generate all combinations of given columns, inserting explicit NA rows where they were implicitly missing
tidyr::fill() last observation carried forward — fill a gap with the most recent non-missing value
str_replace_na() turn NA into the literal string "NA" for display 
# Make implicit missings explicit, then carry the last value forward
stocks |>
  complete(year, quarter) |>     # every year x quarter now has a row (NA where absent)
  fill(price)                    # fill the new gaps with the prior price
Warningna.rm = TRUE is a decision, not a default

Silently dropping missings changes which units you’re averaging over. If the missingness is related to the outcome (the unhappiest customers don’t respond), na.rm = TRUE gives a biased answer that looks clean. The honest move is to know why values are missing before you remove them — exactly the selection concern at the heart of causal inference. (Wickham et al. 2023)

Common pitfalls

WarningWhere this goes wrong

  • Testing x == NA. Always NA; use is.na(). (Wickham et al. 2023)

  • A summary that mysteriously returns NA. One missing input poisoned it; add na.rm = TRUE — but only after deciding that dropping is appropriate. (Wickham et al. 2023)

  • Ignoring implicit missingness. “No row” is invisible to is.na(); use complete() to surface the gaps before they bias a count. (Wickham et al. 2023)

  • Filling blindly with fill(). Last-observation-carried-forward assumes the value persisted; that’s a modelling assumption, not a fact. (Wickham et al. 2023)

## Why it matters

Missingness is not a formatting nuisance; it is a substantive feature of the data with its own statistics.
Where the holes are, and *why*, can bias every estimate downstream — the same selection logic that breaks
[censored survival data](../03-regression/survival-analysis.qmd) or an
[experiment with attrition](../05-causal-inference/randomized-experiments.qmd). R forces you to confront it
rather than letting it pass silently, which is a feature, not an annoyance. [@wickham_r4ds]

## `NA` is contagious

In R, **`NA` represents an *unknown* value.** [@wickham_r4ds] And because it's unknown, almost any operation
touching it is *also* unknown — *"`NA` is contagious."* [@wickham_r4ds]

```r
NA > 5      # NA  — is an unknown number bigger than 5? unknown.
10 == NA    # NA  — does 10 equal an unknown? unknown.
NA == NA    # NA  — are two unknowns equal? STILL unknown (the confusing one)
```

That last line is the trap: **`NA == NA` is `NA`,** not `TRUE`. So you can never test for missingness with
`== NA`; you must use **`is.na()`**. [@wickham_r4ds]

## Explicit vs implicit missingness

A subtle and important distinction: [@wickham_r4ds]

- An **explicit** missing value is *"the presence of an absence"* — a cell flagged `NA`. You can see it.
- An **implicit** missing value is *"the absence of a presence"* — a row that simply **isn't in the table**.
  A respondent-quarter with no record; a product-month with no sales row. You *can't* see it, and it's the
  more dangerous kind because a naive count treats "no row" as if the case didn't exist.

Much of careful data work is **making implicit missings explicit** so you can decide what to do with them.

## Tools

**Summaries propagate `NA` by default** — `mean()`, `sum()`, `median()` return `NA` if *any* input is
missing, which is R insisting you make a choice. Pass **`na.rm = TRUE`** to drop missings before computing.
[@wickham_r4ds] For wrangling:

| Tool | What it does |
|---|---|
| `is.na(x)` | the **only** correct test for missingness (logical vector) |
| `coalesce(x, y, …)` | take the **first non-missing** value across several vectors — fill from a backup |
| `tidyr::complete()` | generate **all combinations** of given columns, inserting explicit `NA` rows where they were implicitly missing |
| `tidyr::fill()` | **last observation carried forward** — fill a gap with the most recent non-missing value |
| `str_replace_na()` | turn `NA` into the literal string `"NA"` for display |

```r
# Make implicit missings explicit, then carry the last value forward
stocks |>
  complete(year, quarter) |>     # every year x quarter now has a row (NA where absent)
  fill(price)                    # fill the new gaps with the prior price
```

::: {.callout-warning title="`na.rm = TRUE` is a decision, not a default"}
Silently dropping missings changes *which units* you're averaging over. If the missingness is related to the
outcome (the unhappiest customers don't respond), `na.rm = TRUE` gives a **biased** answer that *looks*
clean. The honest move is to know **why** values are missing before you remove them — exactly the selection
concern at the heart of [causal inference](../05-causal-inference/potential-outcomes.qmd). [@wickham_r4ds]
:::

## Common pitfalls

::: {.callout-warning title="Where this goes wrong"}

- **Testing `x == NA`.** Always `NA`; use **`is.na()`**. [@wickham_r4ds]

- **A summary that mysteriously returns `NA`.** One missing input poisoned it; add `na.rm = TRUE` — but only
  after deciding that dropping is appropriate. [@wickham_r4ds]

- **Ignoring implicit missingness.** "No row" is invisible to `is.na()`; use `complete()` to surface the gaps
  before they bias a count. [@wickham_r4ds]

- **Filling blindly with `fill()`.** Last-observation-carried-forward assumes the value persisted; that's a
  modelling assumption, not a fact. [@wickham_r4ds]

:::
