Reproducible Reports with Quarto

The last station of the workflow. One document that weaves prose, code, and results — so every number regenerates from the data, and the writing never drifts from the analysis.

An analysis is finished only when someone can act on it — and trust it. Quarto is the tool R for Data Science uses to close that loop: a “unified authoring framework” that weaves prose, code, and results into one source file, so the report regenerates every figure and number from the data each time it’s rendered. (Wickham et al. 2023) The narrative can never silently fall out of sync with the computation, because the computation is the document.

Why it matters

This is the communicate station of the workflow — “absolutely critical,” the authors say, because an insight no one can follow changes nothing. (Wickham et al. 2023) Quarto is also the practical face of open and reproducible science: a copy-pasted screenshot of a result is a claim; a Quarto document that rebuilds the result from data and code is evidence. (This very encyclopedia is built in Quarto.)

Literate programming

Quarto descends from Donald Knuth’s literate programming: the document’s spine is a narrative that explains the what and why of an analysis, with the code that performs it woven alongside. (Wickham et al. 2023) It “integrates your work into a document with commentary.” When you render (the old word is knit), a tool (knitr for R) executes the code into a markdown file of results, which pandoc then turns into the final output. (Wickham et al. 2023) You write thinking and computing together; the machine assembles the artefact.

Anatomy of a .qmd file

A Quarto document has three parts: (Wickham et al. 2023)

  1. YAML header — between triple-dash --- fences at the very top; controls document-wide settings (title, date, output format):
---
title: "Penguin report"
format: html
---

  1. Markdown narrative — the prose spine, written in “a lightweight set of conventions for formatting plain-text files”: # for headings, **bold**, _italics_. (Wickham et al. 2023)

  2. Executable code chunks — a fenced block opening with the language in braces, {r} (or {python}), carrying #| options, then code, and closing with a bare fence. When rendered, the code runs and its output is inserted into the document. A chunk that draws a captioned boxplot would set a label, an echo option, and a fig-cap, then call ggplot(...) + geom_boxplot() — and the report shows the plot, regenerated from the data, not a pasted image. (Wickham et al. 2023)

Chunk options

Each chunk’s behaviour is tuned by options (in the #| header): (Wickham et al. 2023)

Option Effect
echo show the source code? (false → show the result but hide the code)
eval run the code? (false → display it, don’t execute)
include false → run the code but show neither code nor output
warning / message suppress warnings/messages from the rendered output
fig-cap caption a figure (and float it in some formats)
label a unique name — used for graphic filenames and easy navigation/cross-referencing

One source, many formats

The payoff of separating content from format: a single .qmd renders to a “vast array” of outputs without rewriting. (Wickham et al. 2023)

  • Documents — HTML (the default), PDF (via LaTeX), MS Word.
  • Presentations — revealjs / Beamer slides.
  • Bigger structures — dashboards, whole websites, and books (the bookdown lineage).

Write once; deliver as a web page for a colleague, a Word doc for a journal, slides for a meeting.

Why reproducibility is the point

The authors call this “the right way to communicate data analysis” for concrete reasons: (Wickham et al. 2023)

  • In sync. Because every number and figure regenerates from the data on each render, the narrative can’t drift from the computation — no stale copy-pasted table.
  • A definitive record. The document is the analysis; it’s auditable and re-runnable, sparing you the “long-term agony” of finding a result in your workspace but having forgotten the code that made it.
  • Efficient. Caching (cache: true) saves long-running chunks and recomputes only when their code changes.
  • For future-you. It lets you “record your thoughts as you go,” so a collaborator — or you in six months — can reconstruct why each choice was made.

Common pitfalls

WarningWhere this goes wrong

  • Pasting results instead of computing them. A static screenshot breaks the moment the data updates; let the chunk regenerate it. (Wickham et al. 2023)

  • Relying on a hidden workspace. If a number depends on objects not created in the document, the report isn’t reproducible. Keep the analysis self-contained. (Wickham et al. 2023)

  • Showing code your audience doesn’t want. Use echo: false for an executive report; keep it visible for a methods appendix. Match the output to the reader. (Wickham et al. 2023)

  • Forgetting cache invalidation. Caching speeds renders but can serve stale results if you change data the chunk reads without changing its code. (Wickham et al. 2023)
