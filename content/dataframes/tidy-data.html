Tidy Data: One Row, One Observation

  • Show All Code
  • Hide All Code
  • View Source

The single structural convention that makes every later tool just work — and the Tolstoy line that explains why messy data is so much harder.

There are countless ways to arrange the same numbers in a table, and only one that every tool in the ecosystem is built to expect. Tidy data is that one shape: each variable a column, each observation a row, each value a cell. Get your data tidy and the rest of analysis stops being a fight; leave it untidy and you spend your day reshaping instead of thinking.

Why it matters

A dataset’s meaning and its layout are two different things, and beginners conflate them. The same study can be stored a dozen ways — values spread across columns, variables crammed into one column, observations split across tables. Wickham & Grolemund’s insight is that committing to one layout pays compounding dividends: every package in the tidyverse (and most models in this book) assumes it, so a tidy table lets you “focus your struggle on the questions you care about, not fighting to get the data into the right form.” (Wickham et al. 2023)

The three rules

A dataset is tidy if it satisfies three interlocking rules: (Wickham et al. 2023)

  1. Each variable must have its own column.
  2. Each observation must have its own row.
  3. Each value must have its own cell.
Show the figure code 
import matplotlib.pyplot as plt

fig, (axL, axR) = plt.subplots(1, 2, figsize=(8.6, 3.0))

def table(ax, title, col_labels, rows, highlight_cols=None, title_color="#1c2733"):
    ax.set_title(title, fontsize=10.5, color=title_color, fontweight="bold")
    ax.axis("off")
    t = ax.table(cellText=rows, colLabels=col_labels, loc="center", cellLoc="center")
    t.auto_set_font_size(False); t.set_fontsize(9); t.scale(1, 1.5)
    for j in range(len(col_labels)):
        c = t[0, j]; c.set_facecolor("#1f3a5f"); c.set_text_props(color="white", fontweight="bold")
        if highlight_cols and j in highlight_cols:
            c.set_facecolor("#9a6a2f")

# untidy: years as columns
table(axL, "Untidy: headers are values", ["country", "1999", "2000"],
      [["A", "745", "2666"], ["B", "37737", "80488"]], highlight_cols=[1, 2],
      title_color="#9a6a2f")
# tidy
table(axR, "Tidy: headers are variables", ["country", "year", "cases"],
      [["A", "1999", "745"], ["A", "2000", "2666"],
       ["B", "1999", "37737"], ["B", "2000", "80488"]], title_color="#3f8f6f")

fig.tight_layout()
plt.show()

The rules are not arbitrary aesthetics; they line up with how computers and the tidyverse vectorise work — operations run down columns, and “one observation per row” is exactly what a model, a group-by, or a plot iterates over. (Wickham et al. 2023)

The Tolstoy rule

Why does messy data feel endlessly varied while tidy data feels boringly uniform? The authors answer with a borrowed line from Anna Karenina: (Wickham et al. 2023)

“Happy families are all alike; every unhappy family is unhappy in its own way. Tidy datasets are all alike, but every messy dataset is messy in its own way.”

This is the whole argument in one sentence. There is essentially one tidy shape, so once you know it you can learn a single set of tools and apply them everywhere. Messiness, by contrast, is infinite — values in headers, multiple variables jammed in one column, one observation smeared across rows — so ad-hoc fixes never generalise. Investing in the one tidy form is what makes your skills transferable across every future dataset. (Wickham et al. 2023)

Tidying: pivoting

Real data arrives untidy, so tidying is a routine first move, and it usually reduces to pivoting — and the two common problems each have a direction: (Wickham et al. 2023)

  • A variable is spread across column headers (like 1999 and 2000 above). The headers are values of a variable (year), not variable names. Pivot longer: gather those columns into a name column and a value column.
library(tidyverse)

# values trapped in the column names -> gather them into rows
table_untidy |>
  pivot_longer(cols = c(`1999`, `2000`),
               names_to  = "year",
               values_to = "cases")
  • One observation is split across several rows (e.g. cases and population for the same country-year sit on different rows). Pivot wider: spread the name/value pair back into one column each, reuniting the observation onto a single row. (Wickham et al. 2023)
table_long |>
  pivot_wider(names_from  = type,    # "cases" / "population"
              values_from = count)

Pivoting longer and wider are inverses; together they reshape almost any real table into the one tidy form the rest of your tools expect.

Why tidy is the foundation of this book

Tidy data is the silent precondition of nearly every method in this encyclopedia. A regression wants one row per observation and one column per predictor; an exploratory plot maps tidy columns to visual properties; a causal design needs treatment, outcome, and covariates as clean, separate columns. (Wickham et al. 2023) When a modelling function throws a baffling error, the cause is very often an untidy input — a variable hiding in the headers, or two variables sharing a cell.

Common pitfalls

WarningWhere this goes wrong

  • Column headers that are values, not names. 2019, 2020, 2021 as separate columns means year is a variable with no column. Pivot longer. (Wickham et al. 2023)

  • Two variables in one cell. A rate cell holding "745/19987" packs cases and population together; separate them before you can compute anything. (Wickham et al. 2023)

  • One observation across multiple rows. If a single country-year’s cases and population sit on different rows, group-bys and joins misbehave. Pivot wider. (Wickham et al. 2023)

  • Tidying for the machine and forgetting the human. Tidy is the right shape for computing; a final table for readers may deliberately be untidy (wide, with year columns) because it’s easier to scan. The two audiences want different shapes. (Wickham et al. 2023)

- **A variable is spread across column *headers*** (like `1999` and `2000` above). The headers are *values*

  and a `value` column.

```r

# values trapped in the column names -> gather them into rows
```

  country-year sit on different rows). **Pivot wider**: spread the `name`/`value` pair back into one column

```r
table_long |>
  pivot_wider(names_from  = type,    # "cases" / "population"
              values_from = count)
```

the rest of your tools expect.
