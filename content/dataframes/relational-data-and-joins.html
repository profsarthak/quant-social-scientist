Relational Data: Thinking Across Tables

  • Show All Code
  • Hide All Code
  • View Source

Real analysis rarely lives in one table. Keys connect them; joins combine them — and one join should be your default.

Data almost never arrives as a single tidy table. It comes as several — survey responses in one, respondent demographics in another, region codes in a third — linked by shared identifiers. Thinking relationally means seeing those keys and knowing how to join on them. Get joins right and scattered tables become one analysis-ready frame; get them wrong and you silently drop or duplicate observations.

Why it matters

The transform station usually begins by assembling the data, because the information you need is spread across tables — “data analysis rarely involves a single data frame.” (Wickham et al. 2023) A flight has a carrier code; the carrier’s name lives in a separate airlines table. To plot delay by airline name, you must join them. The relational mindset — keys connecting tables — is the difference between an analyst who can only use pre-merged data and one who can build the dataset the question actually requires.

Keys: how tables connect

Tables are linked by keys — variables that identify observations: (Wickham et al. 2023)

  • A primary key uniquely identifies each observation in its own table (e.g. carrier code in the airlines table — one row per carrier).
  • A foreign key is that same variable appearing in another table, where it points back to the first (e.g. carrier in the flights table — many rows per carrier).

A primary key in one table matched to a foreign key in another is a relation. Verifying that your primary key really is unique — exactly one row per value — is the unglamorous check that prevents silent duplication later. (Wickham et al. 2023)

Joins: combining on a key

A join uses the key to bring columns from one table into another, matching rows where the keys agree. R for Data Science pictures a match as a dot at the intersection of a row from each table. (Wickham et al. 2023) The mutating joins add variables from one table to another, and they differ only in which unmatched rows they keep: (Wickham et al. 2023)

  • inner_join() — keep only rows that match in both tables. Unmatched rows are dropped from both.
  • left_join() — keep all rows of the left table, filling NA where the right has no match.
  • right_join() — keep all rows of the right table.
  • full_join() — keep all rows of both, NA wherever either side is missing.
Show the figure code 
import matplotlib.pyplot as plt

fig, (axL, axR) = plt.subplots(1, 2, figsize=(8.8, 3.4))

def draw_join(ax, title, rows, title_color):
    ax.set_title(title, fontsize=10.5, fontweight="bold", color=title_color)
    ax.axis("off"); ax.set_xlim(0, 1); ax.set_ylim(0, 1)
    ax.text(0.28, 0.92, "key  val_x  val_y", fontsize=8, style="italic", ha="center")
    for i, (k, vx, vy, present) in enumerate(rows):
        y = 0.76 - i * 0.18
        color = "#e7f0e9" if present else "#f0e0e0"
        ax.add_patch(plt.Rectangle((0.10, y), 0.55, 0.14, fc=color, ec="#1f3a5f", lw=1.1))
        ax.text(0.37, y + 0.07, f"{k}      {vx}        {vy}", ha="center", va="center", fontsize=9)

# inner: only keys 1,2
draw_join(axL, "inner_join (matches only)",
          [("1", "x1", "y1", True), ("2", "x2", "y2", True)], "#1f3a5f")
axL.text(0.37, 0.04, "rows 3 and 4 dropped", ha="center", fontsize=8, color="#9a6a2f", style="italic")

# left: keep all of x (1,2,3), NA for 3
draw_join(axR, "left_join (keep all of x)",
          [("1", "x1", "y1", True), ("2", "x2", "y2", True), ("3", "x3", "NA", False)], "#3f8f6f")
axR.text(0.37, 0.04, "key 3 kept, y filled NA", ha="center", fontsize=8, color="#3f8f6f", style="italic")

fig.tight_layout()
plt.show()

Why left_join is the default

Wickham & Grolemund give a clear rule of thumb: the left join should be your default. (Wickham et al. 2023) The reason is conservatism — it “preserves all the original observations” in your primary table even when the lookup table has no match. You started with a table of flights; a left join onto airline names keeps every flight, attaching the name where it exists and NA where it doesn’t. An inner join, by contrast, would silently delete any flight whose carrier code was missing from the airlines table — quietly changing your sample. When the left table is your analysis sample, left_join is what keeps that sample intact. (Wickham et al. 2023)

Common pitfalls

WarningWhere this goes wrong

  • Using inner_join as a habit. It drops unmatched rows from both sides; if the left table is your sample, that’s silent, non-random attrition. Default to left_join and check what didn’t match. (Wickham et al. 2023)

  • A non-unique “primary” key. If the key repeats in the table you thought was one-row-per-key, the join duplicates rows — your dataset quietly grows and every later count is wrong. Verify uniqueness first. (Wickham et al. 2023)

  • Mismatched key types or codings. "01" ≠ 1, "USA" ≠ "US"; keys must match exactly to join. Reconcile types and coding before joining. (Wickham et al. 2023)

  • Not inspecting NAs after a left join. The NAs a left join introduces are information — they tell you which rows had no match. Ignoring them hides a data problem. (Wickham et al. 2023)

From the sources

NoteWho says what
  • Wickham & Grolemund, R for Data Science — relational data because “data analysis rarely involves a single data frame”; keys (primary uniquely identifies an observation in its own table; foreign points to it from another); a match pictured as a dot at the intersection; the mutating joins (inner_join keeps only matches; left_join/right_join/full_join keep unmatched rows from one or both sides); and the recommendation that left_join be the default because it preserves all original observations.

Connections

  • Builds on: tidy data (each table tidy, linked by keys) and the grammar of manipulation (joins are verbs that act across tables).
  • Pairs with: the workflow — assembling tables is usually the first move of transform.
  • Underwrites: any analysis whose variables come from multiple sources — merging treatment assignment, outcomes, and covariates into the single tidy frame a model or causal design requires.
