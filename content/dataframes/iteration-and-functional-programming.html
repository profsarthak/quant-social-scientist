Iteration & Functional Programming

Doing the same thing to many columns, files, or groups — without copy-paste and without the bookkeeping of a for-loop. The map family and across().

The companion to writing functions is iteration: applying that function to many inputs — every column, every file, every group. The tidyverse’s answer is functional: across() to do the same thing to many columns, and the purrr map() family to apply a function to each element of a list and get a typed result back. The win over a for-loop is readability — you say what to do, not how to store the results.

Why it matters

Iteration “reduces duplication when you need to perform the same operation on multiple inputs.” (Wickham et al. 2023) It’s the difference between thirty near-identical pasted lines (one per column) and a single expression that can’t drift out of sync. Combined with functions, it’s how an analysis stays short, correct, and reproducible as it scales.

across(): the same thing to many columns

Inside a mutate() or summarize(), across() applies a function to many columns at once — “do the same thing to many columns.” (Wickham et al. 2023) It replaces the older mutate_each()/summarize_each() precursors.

# Summarise the mean of EVERY numeric column, missing values removed — one line, any width
df |>
  summarize(across(where(is.numeric), \(x) mean(x, na.rm = TRUE)))

# Rescale several named columns with our own function, in place
df |>
  mutate(across(c(height, weight, age), rescale01))

The purrr map() family

map() applies a function to each element of a vector or list and returns a result — the functional replacement for a for-loop. (Wickham et al. 2023) Its variants differ only in the type they return, which keeps your output predictable: (Wickham et al. 2023)

Function Returns
map() a list
map_dbl() a double vector
map_int() an integer vector
map_chr() a character vector 
map_dbl(mtcars, mean)          # mean of each column -> a named numeric vector

A canonical use is reading many files: store the paths in a vector and map(paths, read_csv) to apply read_csv() to each, then bind. (Wickham et al. 2023)

files <- list.files("data/", pattern = "\\.csv$", full.names = TRUE)
data  <- map(files, read_csv) |> list_rbind()

Why functional iteration beats loops

For-loops are explicit but verbose: they demand “bookkeeping code” — pre-allocating the output vector, tracking the index — that must be duplicated for every loop. (Wickham et al. 2023) map() hides that machinery, with three consequences the book highlights: (Wickham et al. 2023)

  • Readability — map functions focus on the operation; loops focus on the mechanics of storing results.
  • Composability — they break a list-manipulation problem into independent “bite-sized pieces” that chain with the pipe.
  • Predictable output — the _dbl/_chr/_int suffix guarantees the return type, so a wrong-typed result fails loudly instead of silently.
NoteFunctional, not anti-loop

The point isn’t that loops are forbidden — it’s that the repetitive bookkeeping a loop forces you to re-write is exactly the part that hides bugs. A map() says “this operation, over these inputs” and lets the package handle the rest. (Wickham et al. 2023)

Common pitfalls

WarningWhere this goes wrong

  • Pasting a block once per column. That’s the iteration trigger; use across() so a change happens once. (Wickham et al. 2023)

  • Using map() when you want a vector. map() returns a list; reach for map_dbl/map_chr/map_int to get a typed vector. (Wickham et al. 2023)

  • Re-deriving for-loop bookkeeping. Pre-allocation and indexing are error-prone and repetitive; a functional hides them. (Wickham et al. 2023)

  • Silent type drift. A loop building a result can quietly change type; the typed map_* variants catch it. (Wickham et al. 2023)

## `across()`: the same thing to many columns

## The purrr `map()` family

**`map()`** applies a function to **each element** of a vector or list and returns a result — the functional
replacement for a for-loop. [@wickham_r4ds] Its variants differ only in the **type they return**, which keeps
your output predictable: [@wickham_r4ds]

| Function | Returns |
|---|---|
| `map()` | a **list** |
| `map_dbl()` | a **double** vector |
| `map_int()` | an **integer** vector |
| `map_chr()` | a **character** vector |

```r
map_dbl(mtcars, mean)          # mean of each column -> a named numeric vector
```

A canonical use is [reading many files](importing-data.qmd): store the paths in a vector and `map(paths,
read_csv)` to apply `read_csv()` to each, then bind. [@wickham_r4ds]

```r
files <- list.files("data/", pattern = "\\.csv$", full.names = TRUE)
data  <- map(files, read_csv) |> list_rbind()
```
